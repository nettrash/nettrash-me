# `assets/exchange/` — content for the `exchange.nettrash.me` subdomain

`exchange.nettrash.me` is a one-purpose subdomain whose only job is to host
the Universal Links AASA file for the Exchange iOS / Mac Catalyst app.
The encrypted-envelope URLs Exchange embeds in iMessage bubbles look like:

```
https://exchange.nettrash.me/msg?v=2&p=<base64url-of-encrypted-payload>
```

When a user taps such a URL on iOS or Mac Catalyst, Apple's `swcd` daemon
fetches the AASA below to authorise Exchange (both the iOS bundle ID and
the Catalyst-derived bundle ID) to claim the link. The app then handles
it via `.onOpenURL` and decrypts in place.

Trunk copies the contents of this folder verbatim into
`dist/exchange/`, so they're available under the same dist tree as the
main site. **Nginx is what splits the two hostnames** — see the
"DNS + nginx setup" section below for what's needed to make
`exchange.nettrash.me` actually serve from this subtree.

## Files

### `.well-known/apple-app-site-association`

Apple's Universal Links manifest, scoped to the Exchange app. Must be
served as `application/json` over HTTPS, with no `.json` extension and
no redirects. Apple's `swcd` daemon caches it; expect a propagation
delay of minutes to hours when the file changes.

The single `applinks.details` entry lists both bundle IDs Apple
recognises for Exchange:
- `V4WM2SJ8Q9.me.nettrash.Exchange` — the iOS bundle.
- `V4WM2SJ8Q9.maccatalyst.me.nettrash.Exchange` — the Catalyst-derived
  bundle (auto-generated by Xcode because
  `DERIVE_MACCATALYST_PRODUCT_BUNDLE_IDENTIFIER = YES`).

The single `components` matcher catches `/msg` exactly. The query
parameters carry the actual payload; AASA matches the path independently
of the query so this matcher works for any `?v=…&p=…`.

## DNS + nginx setup

To make `exchange.nettrash.me` resolve and serve this AASA file, two
infrastructure pieces are needed (these are server config, not
build-time things — they live outside the Trunk pipeline):

1. **DNS record.** Add an `A` (or `AAAA`) record for
   `exchange.nettrash.me` pointing at the same IP as `nettrash.me`.
   A `CNAME` to `nettrash.me` works too. Propagation is usually
   minutes via your DNS provider.

2. **nginx server block.** Add a new `server` block to
   `nginx.conf` for `server_name exchange.nettrash.me;` that serves
   the AASA file out of `dist/exchange/.well-known/`. Sketch:

   ```nginx
   server {
       listen 443 ssl http2;
       server_name exchange.nettrash.me;

       ssl_certificate     /etc/letsencrypt/live/nettrash.me/fullchain.pem;
       ssl_certificate_key /etc/letsencrypt/live/nettrash.me/privkey.pem;

       # Universal Links AASA — must serve as application/json with no
       # caching surprises. Apple's swcd refuses anything else.
       location ^~ /.well-known/ {
           root /var/www/nettrash.me/dist/exchange;
           default_type application/json;
           add_header Cache-Control "no-store" always;
           try_files $uri =404;
       }

       # Anything else under exchange.nettrash.me is a pure data carrier
       # (Exchange-encrypted payloads embedded in URL query strings) and
       # has no served content — return 404 so opportunistic crawlers
       # see a hard refusal.
       location / {
           return 404;
       }
   }
   ```

3. **SSL certificate.** Either reissue your existing Let's Encrypt
   cert with `exchange.nettrash.me` as a SAN
   (`certbot --expand -d nettrash.me -d www.nettrash.me -d exchange.nettrash.me …`),
   or use a wildcard cert covering `*.nettrash.me`. The cert path in
   the nginx block above can stay shared with the main site once the
   SAN is added.

After DNS resolves and nginx reloads, verify with:

```bash
curl -i https://exchange.nettrash.me/.well-known/apple-app-site-association
```

You should get `HTTP/2 200`, `Content-Type: application/json`, and the
JSON body. Apple's `swcd` will pick it up on the next AASA fetch (or
sooner if you `sudo killall -INT swcd` on a connected Mac to force a
refresh).
